# RB-11 Disaster Scenario Runbook

## Objective
Restore service while preserving transaction integrity and meeting tier RTO/RPO.

## Roles
- Incident Commander
- Primary On-call SRE
- Database Engineer
- Platform Engineer
- Communications Lead

## Pre-Checks
- Confirm incident scope and severity.
- Capture current health snapshots and lag metrics.

## Numbered Steps (15+)
1. Declare incident severity and assign roles.
2. Freeze risky configuration changes.
3. Run AWS health check query.
4. Validate Route53 health alarms.
5. Confirm regional dependency impact map.
6. Evaluate failover decision threshold.
7. Execute controlled failover trigger.
8. Verify DNS propagation and endpoint health.
9. Promote or validate database writer role.
10. Confirm cache and queue readiness.
11. Run synthetic payment flow tests.
12. Verify reconciliation sanity indicators.
13. Publish stakeholder status update.
14. Decide continue/rollback based on criteria.
15. Stabilize and monitor for 30-minute window.
16. Close incident and capture timeline.
17. Start post-incident review process.

## AWS CLI Commands
1. aws route53 list-health-checks
2. aws route53 change-resource-record-sets --hosted-zone-id ZONE --change-batch file://failover.json
3. aws rds describe-global-clusters
4. aws cloudwatch get-metric-data --metric-data-queries file://queries.json --start-time 2026-07-17T00:00:00Z --end-time 2026-07-17T01:00:00Z
5. aws kafka list-clusters

## Decision Tree (3+ Branches)
- If replication lag > 60s, hold cutover and remediate data path.
- If transaction canary fails twice, rollback DNS and reopen primary flow.
- If reconciliation variance exceeds threshold, halt settlement and escalate to finance controls.

## Verification
- /health endpoints green in promoted region
- Synthetic payment checks pass
- Queue lag below threshold
- Reconciliation mismatch within tolerance

## Rollback
Follow inverse DNS and writer promotion sequence with same verification gates.
